# Supervision tree reshape (precedes the send refactor)

## Why

`Socket` today is a GenServer that `start_link`s its children (ConnectionManager,
SessionManager+ETSKeyStore, EventPublisher) inside `init` and holds their pids in
state — supervisor cosplay with no restart strategy, shutdown ordering, or crash
isolation. The send refactor needs a DynamicSupervisor + Registry for the
per-recipient ConversationSenders, which would pile onto that smell. Fix the tree
first, then do the send refactor under it.

## Target tree (one per connection instance)

```
ConnectionSupervisor (Supervisor, :one_for_one)
├── InstanceRegistry      (Registry, keys = {instance_id, role})  [or shared]
├── EventPublisher
├── ETSKeyStore
├── SessionManager        (depends on ETSKeyStore)
├── ConnectionManager     (login + socket + IQ correlation)
├── SenderSupervisor      (DynamicSupervisor) ── ConversationSender…
└── Socket                (public API GenServer; resolves siblings via Registry)
```

## Public contract — unchanged

`{:ok, handle} = Socket.make_socket(config, opts)` then `Socket.connect(handle)`,
`Socket.send_text(handle, jid, text)`, etc. Today `handle` is the Socket pid.

**Keep it.** `make_socket` starts the `ConnectionSupervisor`, then returns the
**Socket child's pid** (resolved from the tree). All existing call sites
(examples, tests, app code) keep working — they hold a Socket pid and call the
same API.

## Sibling discovery — Registry, replacing the atom-naming hack

Today: `unique_id = rand; name = String.to_atom("connection_manager_#{id}")`.
Replace with a per-instance `Registry`:

- Each child registers under `{:via, Registry, {InstanceRegistry, {instance_id, role}}}`
  where role ∈ `:connection_manager | :session_manager | :event_publisher |
  :sender_supervisor`.
- Socket and ConversationSenders resolve a sibling by role:
  `Registry.lookup(InstanceRegistry, {instance_id, :connection_manager})`.
- `instance_id` = a unique ref/int generated by `make_socket`, passed into every
  child spec. Replaces `unique_id`; gives test isolation without `String.to_atom`
  (which leaks atoms).

Decision to confirm: **one global Registry** (`:unique` keys namespaced by
instance_id) vs **one Registry per instance** (under each ConnectionSupervisor).
Lean: per-instance Registry under the supervisor — cleaner teardown (dies with
the instance), no cross-instance key collisions.

## Startup ordering

`:one_for_one` with children listed in dependency order: ETSKeyStore →
SessionManager → EventPublisher → ConnectionManager → SenderSupervisor → Socket.
Socket last (it resolves the others at init via Registry; they must be up).

The event subscriptions Socket currently sets up in `start_services`
(`ConnectionManager.subscribe(..., self())`) move into Socket's `init` after it
resolves the ConnectionManager pid from the Registry.

## Restart strategy

- `:one_for_one`. A ConnectionManager crash shouldn't nuke SessionManager, etc.
- ConversationSenders are `:temporary` under SenderSupervisor (a failed send is
  not retried).
- Open question: should a ConnectionManager crash restart Socket? Probably the
  whole instance should go down/reconnect as a unit — revisit; for now
  `:one_for_one` and let reconnect logic handle it as today.

## Migration steps

1. Add `ConnectionSupervisor` module (Supervisor) with the child specs +
   per-instance Registry + SenderSupervisor.
2. `make_socket` → start `ConnectionSupervisor`, return the Socket child pid.
3. Socket: drop `start_services`; `init` resolves siblings from Registry, sets up
   subscriptions. `state` keeps the resolved pids (or resolves on demand).
4. Children: accept `instance_id` + register via `:via` Registry.
5. Adapt tests/examples only where they reached into `start_services` internals
   (most just use the public `make_socket`/`connect` and are unaffected).

## Then: send refactor (separate pass, SEND_REFACTOR.plan.md)

ConversationSender starts under SenderSupervisor, resolves ConnectionManager via
Registry, uses `query_iq`/`relay_stanza`. `Socket.send_text` → start_or_lookup
the recipient's sender + cast. ConnectionManager sheds all send logic.

## Open decisions

1. Registry scope: per-instance (lean) vs global namespaced.
2. ConnectionManager crash → restart just it (`:one_for_one`, lean) vs take the
   instance down.
3. Does `make_socket` return the Socket pid (lean, preserves API) or the
   supervisor pid (would break call sites)?
